SSH-Docs

Serve documentation over SSH for developers and AI coding agents.

Python 3.8+ MIT License asyncssh + click

SSH-Docs generalizes the docs-over-SSH pattern: publish markdown documentation as a read-only filesystem that agents can explore with ls, find, grep, cat, head, tail, cd, and pwd. Instead of guessing APIs from stale context, agents can inspect the current source docs directly.

Why SSH-Docs?

Built for terminal-first workflows where humans and AI agents both rely on shell-based exploration.

READ ONLY

Read-only browsing

Expose docs safely with a shell that supports inspection, not modification.

UNIX STYLE

Familiar commands

Use the same primitives agents already use in codebases: grep, find, cat, and ls.

LOW SETUP

Low setup

Run with defaults in minutes, then add YAML config only when you need policy and tuning.

AUTH MODES

Flexible auth

Choose public, password, or SSH key auth depending on whether your endpoint is private or internet-facing.

Quick Start

Install the package, serve a directory, then connect with any SSH client.

pip install ssh-docs

No-Auth Mode (Recommended for Agents)

The simplest way to start is with public (no-auth) mode, ideal for AI coding agents and local development:

# Serve with no authentication required
ssh-docs serve ./docs --auth public

# Or serve current directory with no auth
ssh-docs serve --auth public

# Connect without any credentials
ssh localhost -p 2222

Perfect for AI coding agents, local development, internal networks, and quick documentation sharing.

Basic Usage

# Serve an auto-detected docs directory (uses public auth by default)
ssh-docs serve

# Serve a specific directory
ssh-docs serve ./docs

# Use a custom port
ssh-docs serve ./docs --port 3000

# Connect
ssh localhost -p 2222

AI Agent Workflow

This project extends the public-docs-over-SSH idea to any team: give coding agents direct bash-style access to your docs.

Why it helps agents

Agents frequently fail when they rely on outdated memory: they invent APIs, miss required setup, or apply the wrong CLI flags. SSH-Docs keeps them grounded in your latest markdown content.

How to use with AGENTS.md

Add a short project rule telling agents to check your SSH docs endpoint before making integration decisions.

# AGENTS.md guidance example
Before using project-specific APIs or CLI commands:
1. Connect to docs endpoint: ssh docs.example.com
2. Use find/grep/cat to verify current behavior.
3. Prefer documented commands over assumptions.

Available Shell Commands

The interactive shell intentionally stays small and focused on documentation exploration.

pwd, ls, cd, cat, head, tail, find, grep, help, exit
/docs$ ls
/docs$ cd guides
/docs$ cat getting-started.md
/docs$ grep -R "authentication" .
/docs$ find . -name "*.md"
/docs$ head -n 20 README.md
/docs$ tail -n 10 changelog.txt
/docs$ pwd
/docs$ help
/docs$ exit

CLI Commands

ssh-docs serve

Start the SSH server for a documentation directory.

Options: --port, --site-name, --config, --host, --auth, --keys-file, --password, --no-config, --log-level

ssh-docs init

Create a .ssh-docs.yml configuration file.

ssh-docs validate

Validate config syntax and inspect resolved settings.

ssh-docs keygen

Generate SSH host keys for the server.

ssh-docs completion

Generate shell completion scripts for Bash, Zsh, or Fish.

Configuration Highlights

SSH-Docs can run without config, but a YAML file enables more control.

Core settings

site_name, content_root, port, and host.

Authentication

Set auth.type to public, password, or key.

Features

Feature flags exist for syntax highlighting, line numbers, and search indexing.

Rate limiting

Control connection and failed-auth thresholds in the rate_limiting section.

site_name: "My Project Documentation"
content_root: "./docs"
port: 2222
host: "0.0.0.0"

auth:
  type: "public"

server:
  max_connections: 10
  timeout: 300
  log_level: "info"

features:
  syntax_highlighting: false
  line_numbers: false
  search_index: false

rate_limiting:
  enabled: true
  max_connections_per_ip: 3
  max_connections_per_minute: 10
  max_failed_auth_attempts: 5
  failed_auth_window_seconds: 300
  max_total_connections: 100

Auto-Detection

Content root

SSH-Docs checks docs/, documentation/, public/, and dist/ before falling back to the current directory.

Site name

SSH-Docs reads package.json, then pyproject.toml, and finally the current directory name.
